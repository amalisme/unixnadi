Oracle Linux 8: How to Verify If CVEs Are Patched on Oracle Linux

March 20, 2026 7 min read

Security teams have a habit of sending you a list of CVE IDs and asking, "Are these patched?" It sounds like a simple question, but if you've never dug into Oracle Linux's security tooling before, knowing where to look and how to interpret what you're seeing isn't always obvious.

The good news is that Oracle Linux gives you everything you need through dnf and rpm. No extra agents, no third-party scanners required. This post walks through the commands I actually use when I need to verify CVE status, with notes on what each command tells you — and what it doesn't.

Before You Start: Refresh the Metadata

Before running any security checks, it's worth making sure your local repo metadata isn't stale. This takes a few seconds and ensures you're working with current data:

sudo dnf makecache

It's a small habit that prevents you from drawing wrong conclusions from outdated cache. Now, on to the actual checks.

What's Already Been Patched

The first thing I usually want to know is what security patches are already installed on the system. This command lists all applied security advisories:

sudo dnf updateinfo list installed security

The output lists Oracle Linux Security Advisories (ELSA) that have already been applied, each tied to one or more packages. You'll see something like this:

ELSA-2025-20113 Critical/Sec.  NetworkManager-1:1.40.16-18.0.3.el8_10.x86_64
ELSA-2025-0288  Moderate/Sec.  NetworkManager-1:1.40.16-18.0.1.el8_10.x86_64
ELSA-2021-1679  Low/Sec.       bash-4.4.19-14.el8.x86_64
ELSA-2020-2338  Important/Sec. bind-export-libs-32:9.11.13-5.el8_2.x86_64

The severity column tells you how serious each advisory was — Critical, Important, Moderate, or Low. This command is most useful when you need to prove that patches have been applied, such as during an audit or a compliance review.

Viewing All Patched CVEs

If the security team gives you specific CVE IDs (which is more common), this command maps them directly to installed packages:

sudo dnf updateinfo list installed cves | more

The output looks like this:

CVE-2024-3661    Moderate/Sec.  NetworkManager-1:1.40.16-18.0.1.el8_10.x86_64
CVE-2023-50387   Important/Sec. bind-export-libs-32:9.11.36-11.el8_9.1.x86_64
CVE-2019-18276   Low/Sec.       bash-4.4.19-14.el8.x86_64
CVE-2023-4408    Important/Sec. bind-export-libs-32:9.11.36-11.el8_9.1.x86_64

If a CVE appears here, it means the patch for it has already been applied. This is the most direct answer to "has this CVE been fixed?". I find it more useful than the advisory-based view because security teams almost always think in CVE IDs, not ELSA numbers.

Sorting for Easier Review

When you're dealing with a long list and want to focus on recent vulnerabilities first, pipe the output through sort:

sudo dnf updateinfo list installed cves | sort -nr | more

Checking a Specific CVE

To check a single CVE, filter the output with grep:

sudo dnf updateinfo list installed cves | grep -i CVE-2025-68285

This is the list of CVE-2025-68285 from my VM:

CVE-2025-68285   Important/Sec. perf-4.18.0-553.92.1.el8_10.x86_64
CVE-2025-68285   Important/Sec. python3-perf-4.18.0-553.92.1.el8_10.x86_64

If it shows up, it's patched. If nothing is returned, either the patch hasn't been applied yet, or that CVE simply doesn't affect this system.

⚠️ Common mistake: Don't try to pass a CVE ID directly to dnf updateinfo like this: sudo dnf updateinfo list installed CVE-2025-68285. That syntax doesn't work — dnf updateinfo doesn't accept CVE IDs directly. You need to use grep to filter the full list.

Checking by Advisory ID (ELSA)

Sometimes security teams provide an advisory ID instead of a CVE. In that case, you can query it directly without needing grep:

sudo dnf updateinfo list installed ELSA-2025-0288

And here is the output:

ELSA-2025-0288 Moderate/Sec. NetworkManager-1:1.40.16-18.0.1.el8_10.x86_64
ELSA-2025-0288 Moderate/Sec. NetworkManager-initscripts-updown-1:1.40.16-18.0.1.el8_10.noarch
ELSA-2025-0288 Moderate/Sec. NetworkManager-libnm-1:1.40.16-18.0.1.el8_10.x86_64
ELSA-2025-0288 Moderate/Sec. NetworkManager-team-1:1.40.16-18.0.1.el8_10.x86_64
ELSA-2025-0288 Moderate/Sec. NetworkManager-tui-1:1.40.16-18.0.1.el8_10.x86_64

If the advisory appears in the output, the patch has been installed. This is useful when a vendor or auditor references an ELSA number in their report.

Deep-Dive: Checking a Specific Package Changelog

Sometimes you want more certainty than dnf metadata alone can provide, especially if you're auditing a specific package or suspect the metadata might be incomplete. The package changelog gives you a historical record of every CVE fix that's been applied:

sudo rpm -q --changelog openssl | grep -i cve

The Part Most People Miss

Here's the thing: all the commands above only show what has already been patched. They don't tell you if the system is still vulnerable.

To check for missing patches (eg: CVEs that haven't been applied yet) you need to look at available (not installed) updates:

sudo dnf updateinfo list security | grep CVE-XXXX-XXXXX

Notice the difference: list security without installed shows what's available but not yet applied. If a CVE appears there, the system is still exposed.

To properly answer a security request, you really should check both sides:

  • Search the installed CVE list to confirm if it's been patched
  • Search the available security updates to confirm if it's still pending

From those two checks, the answer becomes clear:

  • Found in installed → it's patched ✓
  • Found in security (available) → still vulnerable, patch not applied ✗
  • Not found in either → likely not applicable to this system

Quick Reference

Here's a summary of all the commands covered in this post:

# Refresh metadata first
sudo dnf makecache

# List all applied security advisories (ELSA)
sudo dnf updateinfo list installed security

# List all patched CVEs
sudo dnf updateinfo list installed cves | more

# Sort CVEs newest-first
sudo dnf updateinfo list installed cves | sort -nr | more

# Check a specific CVE
sudo dnf updateinfo list installed cves | grep -i CVE-2025-68285

# Check by advisory ID
sudo dnf updateinfo list installed ELSA-2025-0288

# Deep check via package changelog
sudo rpm -q --changelog openssl | grep -i cve

# Check if a CVE is still unpatched (available but not installed)
sudo dnf updateinfo list security | grep CVE-XXXX-XXXXX
