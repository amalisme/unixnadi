Why systemctl stop Doesn't Always Work: Understanding Service Dependencies on Oracle Linux

May 5, 2026 5 min read

If you've ever run systemctl stop on a service and watched it spring back to life moments later, you've bumped into one of systemd's most useful and most confusing features: unit dependencies.

In this post, I'll walk through how systemctl exposes these dependencies, why a service might restart on its own, and how to inspect and manage the relationships between units on Oracle Linux.

The Building Blocks: systemd Unit Types

systemd doesn't just manage services. It manages units, and a service is just one type. Here are the most common types you'll encounter:

Unit Type Purpose Example
.service A daemon or process rsyslog.service
.socket A listening socket that can activate a service rsyslog.socket
.timer A scheduled trigger (systemd's cron) logrotate.timer
.target A grouping of units (like runlevels) multi-user.target
.mount A filesystem mount point var-log.mount
.path Watches a file or directory for changes cups.path

A single "service" you care about is often actually a small constellation of these units working together.

A Real Example: Why rsyslog Came Back from the Dead

I ran into this recently while following a vendor KB to clear a full /storage/log partition. The instructions said:

service syslog stop

It "worked" but then I saw this warning:

Warning: Stopping syslog.service, but it can still be activated by:
  syslog.socket

I ignored it and checked the status a few seconds later:

systemctl status rsyslog
# systemctl status syslog
● rsyslog.service - System Logging Service
   Loaded: loaded (/lib/systemd/system/rsyslog.service; enabled; vendor preset: enabled)
   Active: active (running) since Mon 2026-05-04 00:55:24 +08; 10s ago
     Docs: man:rsyslogd(8)
           https://www.rsyslog.com/doc/
 Main PID: 6231 (rsyslogd)
    Tasks: 5 (limit: 4915)
   Memory: 83.8M
   CGroup: /system.slice/rsyslog.service
           └─6231 /usr/sbin/rsyslogd -n

It was running again. Why?

Because rsyslog.socket is socket-activated. The socket unit listens on /run/systemd/journal/syslog, and the moment any process writes a log message to it, systemd automatically starts rsyslog.service to handle it. My own commands were generating audit traffic, which woke the socket, which restarted the service.

This is a feature, not a bug. It's how systemd lazy-loads services on demand. But it means stopping the service alone isn't enough if you need the daemon truly off.

The fix: stop the activator first, then the service.

# systemctl stop rsyslog.socket
# systemctl stop rsyslog.service

Inspecting Dependencies with systemctl

systemd ships with several commands to make these relationships visible. Here are the ones I reach for most often.

See What a Unit Depends On

# systemctl list-dependencies rsyslog.service

This shows everything the unit requires before it can start - a tree of targets, sockets, mounts, and other services.

See What Depends on This Unit

The reverse is often more interesting: who needs me?

# systemctl list-dependencies rsyslog.service --reverse

If you're thinking about stopping a service, run this first. It tells you what else might break.

Show the Raw Dependency Directives

# systemctl show rsyslog.service | grep -E '^(Requires|Wants|After|Before|BindsTo|TriggeredBy)='

You'll see directives like:

  • Requires= — hard dependency; if this fails, we fail
  • Wants= — soft dependency; nice to have, but not fatal
  • After= / Before= — ordering only, not a dependency
  • BindsTo= — even stronger than Requires; if the other unit stops, we stop
  • TriggeredBy= — this is the one that bit me; lists the sockets/timers/paths that can activate this service

If you see TriggeredBy=rsyslog.socket, you know stopping the service alone won't be enough.

A Simple Worked Example

Let's build some intuition with something smaller than a syslog daemon. Most Oracle Linux systems have cups (the printing service) installed, which is also socket-activated.

# What activates cups?
systemctl show cups.service | grep TriggeredBy
# TriggeredBy=cups.path cups.socket

# Stop just the service
systemctl stop cups.service

# Try to print something (or just open a print dialog)
# ...cups.service starts itself back up because the socket caught the request.

# Stop it properly
systemctl stop cups.socket cups.path cups.service

# Now it stays stopped until you start it again, or reboot.

If you want to stop it across reboots too:

# systemctl disable --now cups.socket cups.path cups.service

The --now flag combines disable (don't start at boot) with stop (stop right now).

The Three States You Should Know

When you ask "is this service running?", there are actually three answers:

Command What It Tells You
systemctl is-active foo.service Is the process running right now?
systemctl is-enabled foo.service Will it start at next boot?
systemctl is-failed foo.service Did it crash recently?

A socket-activated service can be inactive but still come up the next time something pokes its socket. That's not a bug; it's the design.

Practical Tips

A few things I've learned:

  1. Always read the warning text. When systemctl stop says "but it can still be activated by:", that list is your homework.
  2. Stop activators before services. Sockets, paths, and timers should generally be stopped before the service they trigger, otherwise you're playing whack-a-mole.
  3. Use --now when disabling. systemctl disable foo.service only affects boot behavior; the service keeps running until you stop it or reboot.
  4. Check both directions. list-dependencies and list-dependencies --reverse answer different questions. Run both before making changes in production.
  5. systemctl cat foo.service shows you the full unit file with any drop-in overrides. Useful when something behaves differently from what the man page suggests.

⚠️ Common mistake: Running systemctl disable foo.service alone won't stop a running service — it only prevents it from starting at boot. Always pair it with --now (or run a separate stop) when you actually want the service off.

Wrapping Up

systemd's dependency model is more nuanced than the old SysV init scripts it replaced, and that nuance trips people up. But once you internalize that a "service" is often a service plus a socket plus a timer plus a path watcher — all loosely coupled through dependency directives — the warnings start to make sense, and systemctl becomes a much more predictable tool.

Next time a service mysteriously restarts itself, your first move should be:

systemctl show <unit> | grep TriggeredBy

The answer is usually right there.
