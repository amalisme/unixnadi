VMware Aria Operations 8.10: Node Primary Replica Failed to Go Offline

February 5, 2026 6 min read

This is the final part of my Aria Operations troubleshooting series. In Part 1 and Part 2, I fixed filesystem check failures on both LVM and standard filesystems. Now we'll tackle the third issue: the cluster being stuck in a "Going Offline" state.

📝 This is Part 3 of a 3-part series:

  • Part 1: Failed to Start File System Check on /dev/data/db
  • Part 2: Failed to Start File System Check on /dev/disk/by-label/root
  • Part 3: Node Primary Replica Failed to Go Offline (this article)

The Problem

After successfully powering on all three nodes (Primary, Primary Replica, and Data node), I logged into the Admin UI and found something unexpected:

  • Cluster Status: Going Offline
  • Availability Status: Activated
  • Primary Replica's State: Running
  • Primary Replica's Status: Waiting for Analytics

After letting the Data node run for a while, the Cluster Status eventually changed to Failed.

What I Tried First

Before finding the solution, I tried several things that didn't work:

  • Clicked on Force Take Cluster Offline — failed
  • Rebooted all three nodes twice following the proper sequence — still greeted with the same status

After some googling, I found Broadcom KB 375791 which provided the solution.

The Fix: Force Cluster Offline via CLI

Step 1: SSH into the Primary Replica Node

Enable the SSH toggle button in the Admin UI and log in as root to the Primary Replica node.

Step 2: Verify Node Slice Status

Use the vrops-status command to check if the node slice is Online or Offline:

vrops-status

Here's the output I received:

root@localhost [ ~ ]# vrops-status
Slice Online-true
admin Role Enabled-true
        vRealize Operations vPostgres Replication Database (vpostgres-repl) is running (1762).
        Setting timezone to:  Asia/Kuala_Lumpur
vRealize Operations Gemfire Locator (gemfire) is running (2181).
data Role Enabled-true
        vRealize Operations vPostgres Database (vpostgres) is running (3307).
        Setting timezone to:  Asia/Kuala_Lumpur
vRealize Operations Analytics (analytics) is running (3495).
        Setting timezone to:  Asia/Kuala_Lumpur
vRealize Operations Collector (collector) is not running.
        Setting timezone to:  Asia/Kuala_Lumpur
vRealize Operations API (api) is not running.
ui Role Enabled-true
remote collector Role Enabled-false

The key line here is Slice Online-true. The node slice is Online, but we need to set it to Offline first before we can properly take the cluster down.

Step 3: Set Node Slice to Offline

Run this command to change the slice status from Online to Offline:

sed -i -e 's/sliceonline = true/sliceonline = false/' /usr/lib/vmware-vcopssuite/utilities/sliceConfiguration/data/roleState.properties

The command returns no output, which is normal.

Step 4: Verify the Change

Run vrops-status again to verify the slice is now Offline:

vrops-status

You should now see:

root@localhost [ ~ ]# vrops-status
Slice Online-false
Offline Reason-failed to properly bring down cluster. force take cluster down - tm30235
admin Role Enabled-true
        vRealize Operations vPostgres Replication Database (vpostgres-repl) is not running.
        Setting timezone to:  Asia/Kuala_Lumpur
vRealize Operations Gemfire Locator (gemfire) is not running.
data Role Enabled-true
        vRealize Operations vPostgres Database (vpostgres) is not running.
        Setting timezone to:  Asia/Kuala_Lumpur
vRealize Operations Analytics (analytics) is not running.
        Setting timezone to:  Asia/Kuala_Lumpur
vRealize Operations Collector (collector) is not running.
        Setting timezone to:  Asia/Kuala_Lumpur
vRealize Operations API (api) is not running.
ui Role Enabled-true
remote collector Role Enabled-false

Notice that Slice Online-false now appears, along with an "Offline Reason" message.

Step 5: Stop the Main Services

Use this command to fully stop the main services:

service vmware-vcops stop

The command returns no output, which is expected.

Step 6: Verify in Admin UI

Go back to the Admin UI. You should now see:

  • Primary Replica: Not Running and in an Offline state
  • Cluster Status: Offline

Running vrops-status one more time shows the Slice is set to Online again:

root@localhost [ ~ ]# vrops-status
Slice Online-true
admin Role Enabled-true
        vRealize Operations vPostgres Replication Database (vpostgres-repl) is running (14200).
        Setting timezone to:  Asia/Kuala_Lumpur
vRealize Operations Gemfire Locator (gemfire) is running (11355).
data Role Enabled-true
        vRealize Operations vPostgres Database (vpostgres) is running (12138).
        Setting timezone to:  Asia/Kuala_Lumpur
vRealize Operations Analytics (analytics) is running (12325).
        Setting timezone to:  Asia/Kuala_Lumpur
vRealize Operations Collector (collector) is running (12530).
        Setting timezone to:  Asia/Kuala_Lumpur
vRealize Operations API (api) is running (12811).
ui Role Enabled-true
remote collector Role Enabled-false

The Moment of Truth

After the cluster was properly Offline, I was able to Bring Cluster Online without any issues. Everything is working as expected — all adapter instances started collecting data, and I have observed no issues since this incident.

Writing these articles reminded me that I still need to delete the snapshots for these three appliances!

Why Set Slice to Offline First?

You might be wondering: why do we need to set Slice Online to false before stopping the cluster?

Honestly, I'm not entirely sure about the technical reason behind this. The KB doesn't explain it in detail. If you know the answer or have insights from Broadcom support, I'd love to hear about it. Feel free to reach out!
