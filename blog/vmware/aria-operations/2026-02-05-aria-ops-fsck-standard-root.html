VMware Aria Operations 8.10: Failed to Start File System Check on /dev/disk/by-label/root

February 5, 2026 5 min read

This is Part 2 of my Aria Operations troubleshooting series. In Part 1, I covered how to fix the filesystem check failure on LVM filesystems for the Primary Replica node. Now we'll tackle the same type of issue, but on standard filesystems for the Data node.

📝 This is Part 2 of a 3-part series:

  • Part 1: Failed to Start File System Check on /dev/data/db
  • Part 2: Failed to Start File System Check on /dev/disk/by-label/root (this article)
  • Part 3: Node Primary Replica Failed to Go Offline

The Problem

Continuing with the troubleshooting from Part 1, after successfully powering on the Primary Replica node, I attempted to power on the Data node. It failed with this error:

Understanding the Difference

Unlike the Primary Replica node which uses LVM filesystems (/storage/db, /storage/log, /storage/core), the root filesystem is a standard partition filesystem. Here's the mapping:

  • /boot = /dev/sda2 (standard filesystem)
  • / (root) = /dev/sda4 (standard filesystem)

The process is slightly different from the LVM approach, but follows the same Broadcom KB 326392.

The Fix: Manual Filesystem Check on Standard Filesystems

Step 1: Power Off the Appliance

Select Power → Power Off and make sure the appliance is completely off.

Step 2: Enter Single User Mode

Power on the appliance and quickly open the console. When you see the boot screen, press 'e' to enter the GRUB menu.

Navigate to the end of the linux line and add the following:

rw init=/bin/bash fsck.mode=force fsck.repair=yes

💡 Note: I had to add fsck.mode=force fsck.repair=yes because I was unable to enter single user mode on the first attempt. These additional parameters force the filesystem check to run and automatically repair issues.

Press F10 or Ctrl+X to boot in single user mode.

Step 3: Execute Filesystem Check

Run the filesystem check on both the boot and root filesystems:

# Check /boot filesystem
e2fsck -y /dev/sda2

# Check / (root) filesystem
e2fsck -y /dev/sda4

Step 4: Reboot

Once both filesystems have completed the fsck, reboot the system:

reboot -f

Wait for the appliance to power on completely.

Result

After performing the filesystem check on both / and /boot, the Data node was able to power on successfully.

The Next Challenge

With all three nodes now successfully powered on, I logged into the Admin UI expecting everything to be working. Instead, I found another issue:

  • Primary Replica's State: Running
  • Primary Replica's Status: Waiting for Analytics
  • Cluster Status: Going Offline
  • High Availability Status: Activated

This was extremely weird. I had just performed a complete power off and power on sequence. How could the Cluster Status be "Going Offline"?
