VMware vSphere Tags: A Beginner-Friendly Guide to Organizing Virtual Machines (Step-by-Step)

March 21, 2026 6 min read

Tags in VMware vSphere are labels you attach to virtual machines so you can organize, search, and manage them more efficiently. Instead of cramming every detail into a VM name, you keep names clean and let tags describe what each machine actually does. It's a small change in how you think about VMs, but it pays off quickly especially once you start using automation or need to generate reports.

I covered the concepts behind vSphere tags in a previous post if you want some background. This post is the hands-on walkthrough: creating a category, creating a tag, and assigning it to virtual machines in vSphere Client.

📋 TL;DR: Tags live inside categories. You create the category first, then the tag, then assign it to a VM. The section below has a quick refresher on the key concepts before we get into the steps.

Quick Recap: How Tags Work in vSphere

Before creating anything, it helps to understand the structure. A category is the group; a tag is the label inside that group. You always create the category first.

Here's a simple example of how you might design your tag structure:

  • Category: Environment — Tags: Production, UAT, Dev
  • Category: Department — Tags: Finance, HR, IT
  • Category: Application Role — Tags: Web, App, DB

When creating a category, you have to choose its cardinality. That is, how many tags from that category a single VM is allowed to have:

  • One tag (single value) - A VM can only hold one tag from this category. Use this for mutually exclusive properties like Environment (a VM can't be both Production and Dev).
  • Multiple values - A VM can hold several tags from the same category. Use this when a VM might serve more than one role (for example, running both Web and App tiers).

Getting cardinality wrong is one of those things you only regret after the fact, so think it through before you create the category.

Step 1: Create a Tag Category

In this example, I want to tag virtual machines that have replication configured. Specifically, VMs replicated using VMware Cloud Director Availability (VCDA). The first thing I need is a category to put that tag in.

Log in to your vSphere Client, click Menu at the top left, then select Tags & Custom Attributes.

Click Tag Categories, then New Category and fill in the details:

  • Name: Replication
  • Cardinality (Tags per Object): One tag
  • Object Types: Virtual machine

Click CREATE.

Step 2: Create a Tag

Now go to the Tags tab and click New Tag. Fill in the details:

  • Name: VCDA-Replicated
  • Description: (optional — add a note if it helps your team)
  • Category: Replication

Click CREATE.

💡 Note: If you skipped Step 1, you can create a new category directly from this dialog using the Create New Category link. Both paths lead to the same result.

You now have a tag called VCDA-Replicated sitting inside the Replication category. Next, assign it to your VMs.

Step 3: Assign the Tag to a Virtual Machine

Navigate to the VM you want to tag. In the Summary tab, search for this tile, and you'll notice the Tags section is empty.

Click ASSIGN, tick the VCDA-Replicated tag, and click ASSIGN again to confirm.

That's it. The tag now appears on the VM's Summary page.

Viewing All VMs with a Tag

One of the most useful things about tags is being able to list all objects that share one. Go back to Menu → Tags & Custom Attributes → Tags, find your tag, and click on it. You'll see a list of every VM assigned to it.

In my case, I only have three replicated VMs on this vSphere instance, so the list is short. That said, I have another environment with over 100 VMs where I need to apply the same tagging; and doing that by hand through the UI would take forever. In the next post, I'll show how to use PowerCLI to bulk-assign tags, which is much faster once you have more than a handful of machines.

A Few Things Worth Keeping in Mind

Tags are easy to create, which makes it tempting to go overboard. A few principles that help keep things clean:

  • Design before you create. Sketch out your categories and tags on paper or a spreadsheet first. Once VMs start accumulating tags, restructuring becomes painful.
  • Keep it simple. Start with a handful of meaningful categories. You can always add more later. It's much harder to clean up a mess of redundant tags.
  • Think about cardinality early. Getting this wrong isn't the end of the world, but fixing it requires reassigning tags across potentially many VMs.

Tags may look like a minor housekeeping feature, but they become genuinely useful the moment you start doing anything at scale such as automation, reporting, policy enforcement, or just trying to find all the VMs belonging to a particular team. They're worth setting up properly from the start.
