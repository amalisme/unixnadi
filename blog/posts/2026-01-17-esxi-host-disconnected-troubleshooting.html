ESXi Host Disconnected: 6 Quick Checks Before Calling Support

January 17, 2026 8 min read

There's nothing quite like the sinking feeling you get when you open vSphere Client and see one or more of your ESXi hosts showing as grey and unresponsive. Whether it says "Not Responding" or "Disconnected," your first instinct might be to panic and immediately call VMware support.

But here's the thing‚Äîmost ESXi disconnection issues can be resolved in minutes with some methodical troubleshooting. I've dealt with dozens of these situations, and the majority turn out to be simple problems with quick fixes.

Before you spend hours on hold with support, let's walk through six critical checks that will solve most disconnection issues. Understanding what's actually happening is half the battle.

Understanding the Difference: Not Responding vs. Disconnected

First things first‚Äîthese two states look similar in vSphere Client (both show a greyed-out host), but they mean very different things:

Not Responding typically means something external is preventing vCenter from communicating with the host. The host might be fine, but vCenter can't reach it. This is usually a network issue, crashed management service, or hardware problem.

Disconnected means vCenter has deliberately stopped monitoring the host. This is either because someone manually disconnected it, there's a licensing problem, or vCenter tried to reconnect and failed.

Here's a quick reference:

State Primary Cause vCenter Behavior
Not Responding Network failure, hostd/vpxa crash, or host hardware failure vCenter stops receiving heartbeats but continues attempting to monitor
Disconnected Manual action, expired license, or failed reconnection vCenter explicitly stops monitoring and ignores all host services

Now let's get into the troubleshooting steps.

Check 1: Verify Network Connectivity and Port Access

The most common reason for a "Not Responding" host is a communication breakdown. vCenter needs to talk to your ESXi host, and that requires both basic network connectivity and specific ports to be open.

Start simple‚Äîtry pinging the ESXi host from your vCenter server:

ping esxi-hostname.domain.com

If the ping fails, you've got a network or routing problem. Check physical connections, switch configurations, and VLANs.

If ping works, the next step is verifying the management ports aren't blocked. vCenter uses:

  • TCP Port 443 ‚Äî Main management communication
  • UDP Port 902 ‚Äî Heartbeat monitoring

Here's something I've seen catch people off guard: if port 902 is blocked by a firewall, the host will appear connected for a brief moment, then immediately flip back to "Not Responding." That's because vCenter can establish the initial connection on 443, but can't receive heartbeats on 902.

Check 2: Restart Management Agents (hostd and vpxa)

If the network is solid, the management services on the ESXi host itself might have crashed or hung. This happens more often than you'd think, especially after high load or storage latency spikes.

The good news? Restarting these services is safe and doesn't affect running VMs.

You can do this from the Direct Console User Interface (DCUI) if you have physical or remote console access:

  • Press F2 to access the DCUI
  • Navigate to Troubleshooting Options
  • Select Restart Management Agents

Or via SSH (if enabled):

/etc/init.d/hostd restart && /etc/init.d/vpxa restart

This restarts both the host daemon (hostd) and the vCenter agent (vpxa). Wait 2-3 minutes, then check vSphere Client to see if the host reconnects.

Pro tip: If you're restarting services via SSH, use the && operator to chain the commands. This ensures both services restart even if you lose your SSH connection.

Check 3: Look for Resource Exhaustion or Purple Screen

Sometimes a host stops responding because it's encountered a critical kernel error‚Äîthe infamous Purple Screen of Death (PSOD). This is ESXi's equivalent of a Windows blue screen.

If you have access to the physical console, iLO, iDRAC, or IPMI interface, check the screen. A PSOD is unmistakable‚Äîit's purple with diagnostic information and error codes.

If you see a PSOD, the host needs to be rebooted. Make sure to photograph or record the error codes first‚Äîyou'll need them for root cause analysis.

If there's no PSOD but the console is sluggish or frozen, the hostd process might be overwhelmed. This usually happens due to:

  • Extreme storage latency
  • Excessive logging filling up the boot partition
  • Memory pressure from too many VMs

You might see messages like "hostd detected to be non-responsive" in the logs. If hostd is struggling, restarting it (Check #2) often resolves the issue temporarily, but you'll need to address the underlying cause.

Check 4: Validate vCenter Managed IP Address

This one is subtle and catches a lot of people off guard. ESXi hosts send their heartbeats to a specific IP address configured in vCenter Server settings. If this IP is wrong or changed, your hosts will stop checking in.

Here's how to verify it:

  • In vSphere Client, go to Administration
  • Select vCenter Server Settings
  • Click Runtime Settings
  • Check the Managed IP Address

This should be the IP address that ESXi hosts can reach to send heartbeats. If it's set to an unreachable address (common after network changes or vCenter migrations), your hosts are essentially sending heartbeats into a black hole.

I've seen this happen after vCenter was moved to a different subnet without updating this setting. All hosts showed "Not Responding" but were perfectly healthy‚Äîthey just couldn't reach vCenter on the old IP.

Check 5: Verify Storage Connectivity

Storage problems, especially high latency or lost connectivity to critical datastores, can cause ESXi management agents to become unresponsive. If the boot datastore is having issues, hostd might not be able to function properly.

If your host is connected to a SAN (Fibre Channel or iSCSI), check:

  • Physical cable connections (yes, cables get bumped)
  • HBA or network adapter status
  • Storage array health
  • Path status in ESXi

You can check storage path status from the ESXi console or via SSH:

esxcli storage core path list

Look for paths marked as "dead" or devices showing as "offline." High storage latency (consistently over 50ms) can also cause management services to hang.

Check 6: Review Licensing and Try Manual Reconnection

If your host is showing as "Disconnected" (rather than "Not Responding"), licensing is often the culprit. When an ESXi evaluation license expires or a subscription lapses, vCenter automatically disconnects the host.

Check your licensing status:

  • Go to Administration ‚Üí Licensing
  • Look for any hosts showing expired or evaluation licenses
  • Assign valid licenses if needed

If licensing looks good, try a manual reconnection:

  • Right-click the disconnected host
  • Select Connection ‚Üí Connect

If this fails, pay close attention to the error message. It will often tell you exactly what's wrong‚Äîcertificate mismatch, version incompatibility, or authentication failure. That error message is your roadmap for the next troubleshooting step.

Quick Reference: Useful Commands

Here are the commands I use most often when troubleshooting disconnected hosts:

Action Command (via SSH)
Restart All Services services.sh restart
Restart Host Daemon /etc/init.d/hostd restart
Restart vCenter Agent /etc/init.d/vpxa restart
Check Agent Status /etc/init.d/hostd status
View Storage Paths esxcli storage core path list

When to Actually Call Support

If you've worked through all six checks and the host is still disconnected, it's probably time to involve VMware support or escalate internally. Here are signs of serious problems:

  • PSOD with hardware-related error codes
  • Multiple hosts disconnecting simultaneously (likely vCenter or network infrastructure issue)
  • Failed host hardware (memory errors, CPU failures visible in logs)
  • Corrupted system partitions on the boot device

The troubleshooting steps above will resolve most common disconnection issues, but they can't fix failed hardware or major vCenter problems.

Wrapping Up

ESXi host disconnections are stressful, but they're rarely as serious as they first appear. In my experience, about 80% of disconnection issues fall into one of these categories:

  • Network connectivity or firewall blocking
  • Crashed management agents
  • Storage latency causing service hangs
  • Licensing problems

By working through these six checks systematically, you can identify and fix most problems in under 15 minutes‚Äîno support call needed. Just remember to stay calm, check the obvious stuff first, and pay attention to error messages. They're usually trying to tell you exactly what's wrong.
