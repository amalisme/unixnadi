VMware Aria Operations 8.10: Failed to Start File System Check on /dev/data/db

February 5, 2026 7 min read

Last week, I received an incident ticket to resolve an issue with Aria Operations where it wasn't collecting or presenting data. Here's the backstory: a datastore was reported full, so the team performed a storage vMotion. Once the datastore had free space again, we noticed that Aria Operations had stopped collecting and receiving data.

Upon investigation, I found three distinct issues:

  1. Failed to start File System Check on /dev/data/db
  2. Failed to start File System Check on /dev/disk/by-label/root
  3. Primary Replica node failed to go Offline

To keep things manageable, I'm splitting this into a three-part series. This article covers Part 1: fixing the filesystem check failure on /dev/data/db, which is an LVM filesystem.

📝 This is Part 1 of a 3-part series:

  • Part 1: Failed to Start File System Check on /dev/data/db (this article)
  • Part 2: Failed to Start File System Check on /dev/disk/by-label/root
  • Part 3: Node Primary Replica Failed to Go Offline

Environment Overview

Here's the setup for this Aria Operations deployment:

  • 1 Primary node
  • 2 Primary Replica nodes
  • 1 Data node
  • Cluster is Online and High Availability is Activated

Initial Symptoms

To troubleshoot, I logged into the Admin UI at https://<Aria Ops FQDN>/admin. Upon login, I could see that the cluster was already in haywire mode. The State, Objects in Process, and Objects Being Collected were showing in progress and ultimately displaying errors.

I observed that I was unable to Take Cluster Offline and also failed to Force Take Cluster Offline. So I proceeded to shut down the appliances following the proper sequence.

Shutdown and Startup Sequence

Following the Broadcom KB on proper shutdown sequence, I powered off the appliances in this order:

  1. Data node
  2. Primary Replica node
  3. Primary node

For powering on, it's the reverse:

  1. Primary node
  2. Primary Replica node
  3. Data node

The Primary node powered on without any issues. However, here came the first challenge—the Primary Replica node failed to power on with this error:

The Fix: Manual Filesystem Check on LVM

Following Broadcom KB 326392, here's what I did to resolve the issue.

Step 1: Identify the Filesystem Type

First, take note that /storage/db is an LVM type filesystem, not a standard partition filesystem. The same applies to /storage/log and /storage/core—they are all LVM filesystems.

Step 2: Enter Single User Mode

Power off the appliance completely by selecting Power → Power Off. Once the appliance is totally off, power it on and get ready to open the web console quickly.

When you see the boot screen, you need to press 'e' to enter the GRUB menu. You have to be quick, or you'll need to start all over again.

Once in the GRUB menu, use the arrow keys to navigate to the end of the linux line, press space, and type:

rw init=/bin/bash

Press F10 or Ctrl+X to boot in single user mode.

Step 3: Activate the LVM

Run the following commands to activate the logical volumes:

lvchange -ay /dev/data/core
lvchange -ay /dev/data/db
lvchange -ay /dev/data/log

💡 What does this command do?

  • lvchange = logical volume change command
  • -a = activate or deactivate a logical volume
  • y = yes (activate)

So lvchange -ay /dev/data/core activates the logical volume core in the volume group data, making /dev/data/core available for use. Think of it like "switching on" the logical volume so Linux can see and use it.

Step 4: Verify LVM Status

Use lvscan to verify all filesystems are active:

lvscan

💡 Reading the output:

An output like ACTIVE '/dev/data/core' [20.00 GiB] inherit means:

  • ACTIVE = the logical volume is available to the system
  • /dev/data/core = logical volume path
  • [20.00 GiB] = size of the logical volume

Step 5: Create Device Nodes

Use vgscan --mknodes to create device nodes for /dev/mapper devices:

vgscan --mknodes

Step 6: Run Filesystem Check

Now run the filesystem check on all three LVM filesystems:

e2fsck -y /dev/mapper/data-core
e2fsck -y /dev/mapper/data-db
e2fsck -y /dev/mapper/data-log

Step 7: Reboot

Once all three filesystems have completed the fsck, reboot the system:

reboot -f

Wait for the appliance to power on completely.

Result

After following these steps, I was able to power on the Primary Replica node successfully.

However, I faced the same problem with the Data node—but with a different filesystem. Unlike the Primary Replica which uses LVM filesystems, the Data node required a filesystem check on standard filesystems.
